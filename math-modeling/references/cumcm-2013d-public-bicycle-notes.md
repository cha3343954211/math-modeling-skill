# CUMCM 2013D 公共自行车服务系统案例记录

## 触发场景

用户要求完成 2013D「公共自行车服务系统」建模全过程，但题目目录可能只有 `CUMCM2013D.doc` 和 `readme.doc`，缺少题面所说的附件1（20个 Excel 借还车数据）和附件2（站点分布图）。

## 关键经验

1. **先做附件审计并诚信披露**
   - 用题目目录作为唯一项目根目录，先建立标准 `支撑材料/` 结构。
   - 若真实 Excel/地图附件缺失，不要声称读取了真实数据。
   - 论文、README、数据审计记录和邮件正文都要明确说明：当前数值为固定随机种子构造的同结构样本结果，不能替代真实运营结论。
   - 保留真实附件读取接口；若用户后续补入真实附件，可按同一字段口径重算。

2. **数据结构建议**
   - 站点表字段：`站点编号`、`站点名称`、`区域类型`、`经度`、`纬度`、`锁桩数量`。
   - 借还记录字段：`日期`、`借车卡号`、`借车站点`、`还车站点`、`借车时间`、`还车时间`、`用车时长_min`。
   - 固定随机种子；仿真只用于验证建模流程，不用于真实政策结论。

3. **推荐模型链**
   - 问题1：站点-日期双索引分组统计每日/累计借车频次、还车频次；按累计频次排序；用车时长经验分布。
   - 问题2：每日不同借车卡数量 `nunique`；单卡累计借车次数长尾分布。
   - 问题3：最大用车日 `argmax_d trips_d`；用 Haversine 定义站点间距离；排除同站记录找非零最短/最长距离；统计同站借还且时长>1min；按小时统计重点站点借还时刻；把一天划为清晨/早高峰/上午平峰/午间/下午平峰/晚高峰/夜间/深夜，识别各站点借还高峰。
   - 峰型归类：以每站 8 个借车时段频次 + 8 个还车时段频次构成 16 维向量，标准化后 KMeans 聚类；用轮廓系数说明类别清晰度，不把低轮廓系数结果过度解释为硬分类。
   - 问题4/5：设置 baseline 为单纯总流量排序；主模型用“流量—每桩周转强度—借还净流入率绝对值—峰值压力”综合评分，输出扩容/调度/低利用站点建议。

4. **配置评价公式**

```text
q_i = (B_i + R_i) / 天数
u_i = q_i / c_i
rho_i = (R_i - B_i) / (R_i + B_i + eps)
S_i = 0.35*norm(q_i) + 0.25*norm(u_i) + 0.25*norm(|rho_i|) + 0.15*norm(h_i)
```

解释：`q_i` 反映总需求，`u_i` 反映锁桩承载压力，`rho_i` 反映潮汐失衡，`h_i` 反映高峰压力。若 `u_i` 高且 `rho_i` 低，偏向扩容；若 `|rho_i|` 高，偏向分时调度；若 `u_i` 低且总量低，考虑迁移或减少锁桩。

5. **验证与论文写法**
   - L2：冻结 `frozen_numbers.json`，保存所有结果表和图表；检查距离非负、时长范围、锁桩分母非零。
   - L3：摘要必须说明数据附件缺失限制，同时给出模型流程和仿真样本关键数字；不要把仿真数字写成真实温州运营结论。
   - 灵敏度：扰动综合评分权重 ±10%/±20%，比较 Top10 重点站点重合率。

6. **交付细节**
   - PDF 编译后用 `pdfinfo` 核验页数，数学建模正式论文应达到 15 页以上。
   - 邮件发送前核验 zip 内含：`支撑材料/code/main_modeling.py`、`支撑材料/papper/论文.pdf`、`支撑材料/results/frozen_numbers.json`、关键结果表、`readme.txt`。
   - 若附件缺失，邮件正文也要说明“本次为同结构仿真样本完整流程，补入真实附件后可重算”。

## Pitfalls

- 不能把缺失真实附件情况下的仿真结果包装成真实数据分析。
- 不能只按总流量评价锁桩配置；高总量可能只是容量充足的枢纽，高周转/高失衡才是扩容或调度信号。
- 聚类轮廓系数较低时，要写成“调度分组参考”，不要写成天然清晰类别。
