# 三层质量门控：建模合理性、求解正确性、论文质量

> 本参考文件是 `math-modeling` skill 的质量增强模块。每次实际完成建模题、论文或支撑材料时，必须用它做过程门控和最终审计。

## 总原则

数学建模作品的高分不只来自“用了什么算法”，而来自三条链条同时闭合：

1. **建模合理性**：题目需求 → 假设 → 变量/参数 → 模型结构 → 现实解释，逻辑闭合。
2. **求解正确性**：数据 → 代码 → 算法 → 数字结果 → 图表，计算可复现且无明显错误。
3. **论文质量**：模型与结果被清楚、规范、有说服力地写出来，评委能在短时间内抓住贡献。

任何一层不合格，都不能进入最终提交。

## L1 建模合理性门控

### 1. 题目-模型映射

每个子问题都必须先写出一行映射：

```text
子问题 i：题目要求输出 ___；核心决策/预测/评价对象 ___；选择 ___ 模型；输出形式为 ___。
```

检查标准：

- 模型输出必须直接回答题目问题，不能只做“相关分析”却没有给出题目要求的方案/预测/排序。
- 如果题目要求“给出方案”，论文中必须有可执行方案，而不是只给模型评价指标。
- 如果题目要求“解释原因”，不能只给黑箱预测，必须补充变量贡献、敏感性或机制解释。

### 2. 假设合理性审查

每条假设必须满足三项：

| 检查项 | 合格标准 | 不合格表现 |
|---|---|---|
| 必要性 | 去掉该假设，模型会明显变复杂或无法求解 | 写“数据真实可靠”等空泛假设凑数 |
| 合理性 | 与题意、常识、数据范围不冲突 | 为得到想要结论而假设结果 |
| 可追踪性 | 后续公式/算法确实用到了该假设 | 假设列了但正文完全没用 |

写作格式建议：

```text
假设1：……。合理性：……；作用：该假设用于简化/确定……，对应模型中的……。
```

### 3. 变量、参数、量纲一致性

建模完成后做一次“量纲走查”：

- 决策变量、状态变量、参数、常数分开定义。
- 所有符号在符号表中只出现一次，含义不变。
- 目标函数左右量纲一致；约束两侧量纲一致。
- 指标归一化前后含义清楚，正向/逆向指标不混淆。
- 单位换算写在数据预处理或模型建立中，不能藏在代码里。

### 4. Baseline 与主模型必要性

每个主模型默认要有 baseline。baseline 可以很简单，但必须能回答同一个问题。

| 题型 | 合格 baseline 示例 |
|---|---|
| 预测 | 上一期、移动平均、线性回归、灰色预测 |
| 优化 | 当前方案、均分方案、贪心、小规模穷举 |
| 评价 | 等权评分、单指标排序 |
| 分类 | 多数类、简单阈值、KMeans 初版 |
| 仿真 | 简化规则仿真、无干预情景 |

主模型必须说明“为什么比 baseline 更适合”：精度更高、约束更完整、解释更强、方案更可执行，至少占一项。

### 5. 现实可解释性

论文中至少回答：

- 模型结果是否符合常识？若不符合，原因是什么？
- 参数取值是否有来源？来自题目、文献、估计、校准还是假设？
- 约束是否覆盖现实硬约束？如容量、非负、时间、预算、守恒、边界条件。
- 模型忽略了哪些因素？这些忽略是否会改变主要结论？

### L1 不通过时的修复

- 输出不对题：回到问题分析，重写“输出形式”和目标函数。
- 假设牵强：删除无用假设，改为数据限制或模型局限。
- 模型过复杂但解释不了：降级为可解释模型，复杂模型放作对比或改进。
- 没有 baseline：先补最简单可运行 baseline，再比较。

## L2 求解正确性门控

### 1. 数据正确性

正式建模前必须生成数据审计摘要，至少包含：

```text
数据来源：___
样本量/字段数：___
缺失值：___
异常值处理：___
单位换算：___
训练/测试或时间切分规则：___
可能的数据泄漏风险：___
```

禁止做法：

- 先随机打乱时间序列再做预测划分。
- 用全量数据标准化后再划分训练/测试。
- 删除异常值但不说明规则和数量。
- 把题目要求预测的未来信息用于训练。

### 2. 算法与代码审查

每个关键脚本必须满足：

- 可从项目根目录或对应 `questN/codes/` 独立运行。
- 固定随机种子：`np.random.seed(42)` / `random_state=42`。
- 输入文件、输出文件路径清楚，不依赖本地临时路径。
- 关键中间结果保存到 `outputs/`，图表保存到 `figures/`。
- 可视化必须 `plt.savefig(..., dpi=300, bbox_inches='tight')` + `plt.close()`，不要 `plt.show()`。
- 对优化问题，输出目标值、变量值、约束残差/是否满足约束。
- 对预测/分类问题，输出训练/测试指标，不只输出训练集效果。

### 3. 数字结果复核

结果进入论文前必须冻结并复核：

1. 将最终数字写入 `frozen_numbers.json` 或 `outputs/final_results.json`。
2. 论文、摘要、图表标题中的数字只引用冻结文件。
3. 随机算法至少重复运行 3 次，检查结论是否稳定；不稳定则报告均值±标准差。
4. 重要公式用小样本手算或 NumPy/SymPy 代入验证。
5. 对优化结果检查约束满足情况；对预测结果检查误差量级是否符合原数据单位。

### 4. 分类验证清单

| 题型 | 必做验证 |
|---|---|
| 预测 | 时间切分/训练测试划分、MAE/RMSE/MAPE、残差图、与 baseline 对比 |
| 优化 | 可行性检查、约束残差、目标值对比、关键参数敏感性、小规模最优校验 |
| 评价 | 指标方向、权重来源、一致性检验/熵权计算、排名稳定性、等权对比 |
| 聚类 | 标准化、轮廓系数/CH 指数、不同 K 或参数对比、簇画像解释 |
| 分类 | 混淆矩阵、Precision/Recall/F1、类别不平衡处理、交叉验证 |
| 机理/微分方程 | 初值/边界条件、参数物理意义、量纲、守恒或趋势合理性 |
| 仿真 | 随机种子、重复实验、置信区间、情景参数来源 |

### 5. 错误侦测信号

遇到以下信号必须暂停，不得继续写论文：

- 结果数量级离谱，如概率不在 `[0,1]`、负库存、超过容量。
- 图表趋势和文字结论相反。
- 代码重复运行结果变化很大但未说明随机性。
- 同一数字在摘要、正文、表格中不一致。
- 模型指标“过好”：R²≈1、误差≈0，要检查数据泄漏或目标变量混入特征。
- 优化器返回失败状态仍使用结果。

### L2 不通过时的修复

- 先保存错误现象和最小复现，不要直接改论文数字。
- 回到代码检查数据读取、排序、单位、索引对齐、merge 后行数变化。
- 用小规模数据手算或穷举验证算法。
- 重跑所有输出，并重新生成冻结数字和图表。

## L3 论文质量门控

### 1. 摘要质量

摘要必须包含：

- 每个子问题的模型名称或核心方法。
- 每个子问题至少一个关键数值结果或明确结论。
- 模型检验/灵敏度分析的一句话结果。
- 不出现“效果较好”“具有一定意义”这类无数字支撑的空话。

摘要检查公式：

```text
摘要 = 问题背景 + 子问题1方法和结果 + 子问题2方法和结果 + 子问题3方法和结果 + 稳健性/推广结论
```

### 2. 逻辑链完整性

正文每个子问题按固定链条写：

```text
问题要求 → 分析难点 → 假设/变量 → 模型公式 → 求解方法 → 结果表/图 → 结果解释 → 检验/对比
```

不允许：

- 先给结果，后补模型。
- 只有代码输出，没有公式和解释。
- 公式很多但没有说明每个公式服务什么问题。
- 图表堆叠，没有“由图/表可知……”的分析文字。

### 3. 图表与公式质量

- 表题在上，图题在下，编号连续。
- 所有图表必须在正文中引用并解释。
- 图表坐标轴有单位，颜色/图例可读，分辨率 ≥300 dpi。
- 公式编号连续，重要公式后解释变量和实际含义。
- 图表不直接截图 Excel；优先用 Python/LaTeX 生成清晰图表。

### 4. 结果表达质量

每个关键结果使用“数值 + 对比 + 含义”三段式：

```text
模型得到 ___ = 12.36，相比 baseline 的 15.82 降低 21.9%，说明 ___。
```

不要只写：

```text
模型效果较好，结果合理。
```

### 5. 模型评价与局限

模型评价不能只写模板话，必须与本文模型对应：

- 优点：对应本文实际做了什么，如“引入容量约束”“考虑季节性”“使用组合赋权降低主观性”。
- 缺点：对应本文未解决什么，如“未考虑突发天气”“参数来自历史均值，对极端情景适应不足”。
- 改进：逐条回应缺点，不写泛泛的“未来可以优化算法”。

### 6. 最终排版与提交

- 中文论文正式版优先 `.tex` + XeLaTeX 编译，不用 pandoc 直接转 PDF。
- 参考文献格式统一，正文必须有引用标注。
- 附录代码不宜过长，核心代码完整，冗余日志删除。
- 支撑材料目录按 `papper/ quest1/ quest2/ quest3/ readme.txt` 组织。

### L3 不通过时的修复

- 摘要无数字：从冻结数字中补齐各子问题核心结果。
- 逻辑断裂：按“问题要求→模型→求解→结果→解释”重排小节。
- 图表无解读：每张图表后补 2-4 句分析。
- 论文像代码报告：增加数学公式、变量定义、假设合理性和模型评价。

## 三层最终审计表

提交前逐项打勾：

```text
【L1 建模合理性】
□ 每个子问题都有“题目要求 → 模型输出”的映射
□ 每个主模型都有 baseline 或明确说明无需 baseline
□ 假设必要、合理，并在模型中被使用
□ 决策变量/参数/单位/约束定义清楚
□ 结果有现实解释，异常结论有原因说明

【L2 求解正确性】
□ 数据审计完成：缺失、异常、单位、划分、泄漏风险已说明
□ 代码可复现，路径规范，随机种子固定
□ 优化约束/预测误差/评价权重/聚类稳定性等验证已完成
□ 关键数字已冻结，论文所有数字与冻结文件一致
□ 图表由最终代码生成，未手工改数

【L3 论文质量】
□ 摘要覆盖所有子问题，含具体方法和数字结果
□ 每个子问题都有公式、求解、结果、解释、验证
□ 所有图表编号、标题、单位、正文引用齐全
□ 模型评价具体对应本文方法，不是通用套话
□ LaTeX/PDF、参考文献、附录、支撑材料目录均符合要求
```

## 输出给用户时的工作习惯

当用户让你实际做数学建模题时，回复和文件交付应体现三层质量状态：

```text
已完成：
- L1 建模合理性：通过/待补，原因……
- L2 求解正确性：通过/待补，验证……
- L3 论文质量：通过/待补，改进……
```

不要只说“已经完成论文/代码”，要说明三层门控是否通过。
