# CUMCM 2011D 天然肠衣搭配问题踩坑记录

## 任务类型

天然肠衣搭配问题属于离散装箱/切割库存/整数规划问题。原料长度按 0.5m 分档，成品要求：

- 3--6.5m 规格：20 根，允许 19 根；
- 7--13.5m 规格：8 根，允许 7 根；
- 14m 以上规格：5 根，允许 4 根；
- 每捆总长度 89m，允许 `[88.5, 89.5]`；
- 长原料可降级使用，短原料不能升级；
- 目标为字典序：先最大总捆数，再最大长规格捆数，再最大中规格捆数。

## 本次暴露的关键问题

### 1. 工作目录必须一开始就在题目目录内

用户明确纠正：不要把中间文件放到题目目录外的临时目录。做真题练习时，必须先在题目目录下创建标准 `支撑材料/` 结构，再开始提取题面、写代码和生成结果。

正确路径示例：

```text
<WORKSPACE>/Hermes Agent Communityworks/mathmodel/test/2011年赛题/2011D/支撑材料/
├── papper/
├── quest1/codes/ figures/ outputs/
├── quest2/codes/ figures/ outputs/
├── quest3/codes/ figures/ outputs/
├── data/
├── references/
├── results/
├── tables/
└── readme.txt
```

若为方便 shell 避免中文路径而创建了临时目录，必须只作为短暂 scratch，且正式产物立即复制回题目目录；用户要求继续时要清除 scratch。

### 2. 聚合整数规划会产生“伪可行上界”

对 2011D，聚合模型容易写成：

- 对每个规格统计总根数；
- 对每个规格统计总长度；
- 限制总根数在 `标准根数*n` 与 `(标准根数-1)*n` 之间；
- 限制总长度在 `88.5*n` 与 `89.5*n` 之间。

这类模型可能很快给出高上界，例如总捆数达到理论长度上界，但它没有保证这些原料能被拆成每一捆都合法的具体配方。因此不能把聚合解直接写入论文或表格。

**必须增加可分解性验证**：最终方案必须落到逐捆表，每一行都能检查：

```text
单捆根数 ∈ {标准根数, 标准根数-1}
单捆总长度 ∈ [88.5, 89.5]
每根原料总使用次数 ≤ 库存
规格下限满足，降级规则满足
```

### 3. 优先使用“可行单捆模式 + 主整数规划/列生成”

比一次性 CP-SAT 给全部 190+ 捆分配更稳的建模路线：

1. 先生成合法单捆模式：每个模式记录规格、各长度档用量、根数、总长度；
2. 用整数变量表示每种模式采用次数；
3. 主问题约束各长度档使用量不超过库存；
4. 目标按字典序加权：`大权重*总捆数 + 次权重*长规格 + 中权重*中规格`；
5. 若全模式枚举爆炸，使用列生成/限制模式池：
   - L、M 规格根数少，可优先全枚举或大量枚举；
   - S 规格 19/20 根，组合爆炸，需动态生成、限制模式池或分阶段处理；
6. 求得模式解后展开为逐捆 `final_matching_plan.csv`，再做验证脚本。

### 4. 不要过早承诺“马上完成”

这类题的难点在可验证的逐捆方案，而不是论文写作。若精确分解未跑出结果，应明确说当前处于模型求解阶段，不要进入论文定稿，也不要生成看似完整但未验证的 PDF。

### 5. 求解器状态与超时排障规则

处理 2011D 这类逐捆分解模型时，CP-SAT/CBC 输出要严格区分：

- `INFEASIBLE`：该固定规格组合在当前逐捆约束下被证明不可行，可以排除；
- `UNKNOWN`：时间限制内未证明可行/不可行，不能当作不可行，更不能当作可行；
- `exit code -15` 或后台任务无输出：进程被终止/超时杀掉，说明模型规模或日志输出不可控，不是数学结论；
- 大量固定组合连续 `INFEASIBLE/UNKNOWN`：说明路线陷入组合爆炸，应停止盲扫，改成“合法单捆模式池 + 主整数规划/列生成/受限模式池”。

遇到长时间无结果时，不要连续启动多个全量 CP-SAT 任务。应先杀掉旧进程，记录已证明不可行的组合，再降维：

1. 只生成单捆合法模式并统计模式数量；
2. L 规格可枚举，M/S 规格优先限制模式池或随机/启发式生成候选模式；
3. 主整数规划先找高质量可行解，再逐步扩大模式池证明或改进；
4. 每次只让一个长任务运行，设置明确 timeLimit，并保存中间可行解；
5. 最终只接受通过验证脚本的 `final_matching_plan.csv`，不能把 `UNKNOWN` 或聚合上界写入论文。


### 4. 模式枚举必须受控：避免 MPS/内存爆炸

本题的“可行单捆模式 + 主整数规划”路线是正确方向，但不能天真地一次性全枚举所有模式后交给 PuLP/CBC。一次实测中模式量级达到：

```text
L 约 4,286 个
M 约 825,054 个
S 约 110,001 个
```

随后 PuLP 在写 MPS 文件阶段触发 `MemoryError`，说明瓶颈不一定在求解器搜索，而可能在“模型文件生成/变量列数”阶段。未来处理同类装箱/搭配题时应遵守：

1. **先估算模式数量**：正式建主问题前，先对各规格合法单捆模式计数；若某类超过 1--5 万列，禁止全量进入 PuLP/CBC。
2. **限制模式池**：按长度接近 89m、消耗稀缺库存少、根数标准优先、降级材料少等规则筛选 Top-K 模式。
3. **分批迭代/列生成**：先用小模式池求可行解，检查剩余库存和影子价格/缺口，再补充针对性模式；不要一次性把所有 M/S 模式塞入主问题。
4. **优先保留可审计逐捆解**：即使模式池解不是严格最优，也必须展开为逐捆表并通过根数、长度、库存、降级规则验证，论文中明确写“可行解/近似解/上界差距”。
5. **求解器日志要分辨阶段**：`MemoryError` 多发生在建模或 MPS 写出阶段；`UNKNOWN` 是时间内未证明；`INFEASIBLE` 才是严格不可行。不要把三者混为“无解”。

### 5. 聚合上界需要逐规格分解反证

对 2011D 的一次聚合上界实测，SciPy/HiGHS 聚合整数模型可快速得到：

```text
L=134, M=44, S=16，总计 194 捆
```

但把该聚合分配拆成逐捆后发现：

```text
L 134 INFEASIBLE
M 44 OPTIMAL
S 16 OPTIMAL
```

这说明 `194` 是典型伪可行上界：总量层面满足长度/根数，但长规格无法拆成每捆合法组合。未来论文写作中必须区分：

- **理论长度上界**：如 `floor(总长度 / 88.5)`；
- **聚合整数规划上界**：可用于证明“最多不超过某值”，但不能当生产方案；
- **逐捆可行方案**：只有通过逐捆分解验证后才能作为实际方案；
- **反证记录**：若某个聚合上界逐捆不可分解，应在方法分析中说明它只是上界，不写成最优解。

### 6. 不要用极短视贪心作为最终结果

一次简单逐捆贪心虽然能快速生成合法表，但只得到约 `L=45, M=0, S=0` 的低质量方案。它的价值是验证输出管线（`final_matching_plan.csv`、`material_usage_by_length.csv`、`frozen_numbers.json`）和逐捆检查脚本，不适合作为最终论文结果。若使用贪心，应定位为 baseline，并继续用受限模式池/迭代主问题改进。

## 交付前必须有的验证文件

- `tables/final_matching_plan.csv`：逐捆方案；
- `tables/material_usage_by_length.csv`：各长度档库存、各规格使用量、剩余量；
- `tables/summary_by_spec.csv`：各规格捆数和长度统计；
- `results/frozen_numbers.json`：论文数字冻结；
- 一个验证脚本或验证段落，确认每捆长度/根数/库存/降级规则全部满足。

## 论文写法提示

若最终只得到高质量可行解而未严格证明最优，论文中必须区分：

- 理论上界：如 `floor(总长度/88.5)`；
- 聚合整数规划上界：仅作为上界，不等于可行方案；
- 逐捆可行方案：可作为实际生产方案；
- 最优性声明：只有逐捆方案达到已证明上界时才能写“最优”。
